# 1. *Project Overview*

## 1.1 *Project Title*
**MiniDB - Design and Implementation of a Mini Relational Database Engine from Scratch**

## 1.2 *Problem Statement*
Most student projects use an already available database and build features on top of it.  
Our goal is different: we are building the database engine itself to understand core DBMS internals like storage, indexing, query parsing, and execution.

## 1.3 *Project Objective*
Build a simplified relational database engine that can:
- create and manage tables
- store and fetch records from disk
- parse and execute basic SQL-like queries
- use B+ Tree indexing for faster search
- gradually move to page-based storage and buffer pool architecture

## 1.4 *Scope*
This project includes:
- table creation, listing, and deletion
- data insertion and record search
- metadata management
- query parsing and filtering
- B+ Tree indexing
- storage redesign with pages, RID, and slotted pages
- buffer pool planning and future implementation
- optional API/frontend extensions for usability

# 2. *Current System State and Evolution*

## 2.1 *Current Working Capabilities (Already Available)*
- menu-driven CLI database interaction
- show/create/drop table operations
- insert row functionality
- basic search with primary key and non-primary key paths
- SQL-like select handling (`SELECT`, basic `WHERE`)
- metadata file handling
- B+ Tree based indexed lookup for primary key path

## 2.2 *Legacy Internal Model (Old Path)*
- table metadata stored separately
- records historically tied to per-row file flow
- indexed lookup linked to record-number style access

## 2.3 *New Foundation Added (Issue 2 Work)*
The project now has a storage foundation for DBMS-style internals:
- `RID (page_id, slot_id)` as record address
- `PageHeader` and `SlotEntry` structures
- `DiskManager` for page-level allocate/read/write
- `DataPage` for slotted-page tuple placement
- `TupleSerializer` for value <-> bytes conversion

## 2.4 *Why This Evolution Matters*
- clean path toward buffer pool integration
- clean path toward `key -> RID` indexing
- cleaner separation of storage, indexing, and execution layers
- scalable architecture compared to ad-hoc row file handling

# 5. *System Architecture*

## 5.1 *Layered View*
1. **CLI / Query Input Layer**  
   Accepts user commands and SQL-like statements.
2. **Parser Layer**  
   Tokenizes/parses query statements into execution-ready form.
3. **Execution Layer**  
   Routes query to scan path or index path.
4. **Index Layer (B+ Tree)**  
   Handles key-based lookup and index traversal.
5. **Storage Layer**  
   Handles metadata, tuple bytes, pages, RID mapping.
6. **Disk Layer**  
   Handles physical read/write of pages.
7. **(Planned) Buffer Pool Layer**  
   Caches pages in memory using LRU policy.

## 5.2 *Key Data Flow*
`Query -> Parse -> Plan/Route -> (Index scan OR Sequential scan) -> Storage Access -> Result`

## 5.3 *Storage Path (New Foundation)*
`Row values -> TupleSerializer -> DataPage insert -> slot_id -> RID -> DiskManager page write`

# 6. *Data Design and Internal Structures*

## 6.1 *RID Definition*
`RID = (page_id, slot_id)`

Purpose:
- uniquely locate a tuple inside page-based storage

## 6.2 *Page Header*
Contains:
- page identifier
- slot count
- free space pointers
- page type

## 6.3 *Slot Entry*
Contains:
- tuple offset
- tuple length

## 6.4 *Slotted Page Layout*
- header at front
- slot directory grows forward
- tuple data grows backward
- free space remains in middle

## 6.5 *Tuple Serialization*
Current support:
- `INT`
- `VARCHAR`

Purpose:
- convert logical row values to bytes for storage
- convert bytes back to values for reads

# 7. *Technology Stack and Usage*

| Technology | Usage in Project | One-line Purpose |
| --- | --- | --- |
| **C++ (Core)** | Main engine implementation | Used to build storage, indexing, parsing, and execution internals |
| **STL (vector/map/etc.)** | Internal data structures | Used for efficient in-memory handling of records, nodes, and metadata |
| **File I/O (`fstream`, `FILE*`)** | Disk persistence | Used to store metadata, pages, and table data on physical disk |
| **Make + GCC/G++** | Build system | Used to compile and link the MiniDB engine |
| **B+ Tree (custom)** | Index layer | Used for faster key-based lookup |
| **Slotted Page model** | Storage layer | Used to store multiple tuples inside fixed-size pages |
| **RID addressing** | Record location model | Used to locate tuples as `(page_id, slot_id)` |
| **Tuple Serializer (custom)** | Storage encoding layer | Used to convert row values into bytes and back |
| **Buffer Pool (planned)** | Memory caching layer | Will reduce disk I/O by caching frequently used pages |
| **Node.js / C++ sockets (planned)** | API layer | Will expose DB operations externally |
| **React (planned optional)** | Frontend demo | Will provide UI-based query interaction |
